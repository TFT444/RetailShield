## Workflow

RetailShield uses a **fork → branch → PR** model.

1. Fork the repository to your own GitHub account.
2. Clone your fork locally.
3. Create a branch off `dev` (never off `main`).
4. Make your changes and commit with clear messages.
5. Push your branch and open a pull request targeting `dev`.
6. Wait for review. All PRs are reviewed by the maintainer before merge.

`main` is the stable release branch. All development happens on `dev` and feature branches. Direct pushes to `main` or `dev` are not accepted.

## Branch naming

| Prefix | Use for |
|--------|---------|
| `feat/` | New features or modules |
| `fix/` | Bug fixes |
| `docs/` | Documentation only changes |
| `test/` | Test additions or corrections |

Examples: `feat/loss-prevention-module`, `fix/compliance-timer-overflow`, `docs/sentinel-deployment-guide`

Keep names lowercase, hyphen-separated, and descriptive enough to understand without reading the diff.

## No AI attribution

Do not include AI tool signatures, co-author lines, or generation notices anywhere in this repository:

- No `Co-Authored-By: Claude` or similar in commit messages
- No `Generated by [tool]` in PR descriptions, file headers, or comments
- No AI tool names in branch names

Commit messages and PR descriptions should describe the change itself — what was done and why — not how it was produced.

## Adding a KQL detection rule

Detection rules live in `detection-rules/retail/`. Each rule is a standalone `.kql` file.

### File naming

Use lowercase, underscore-separated names that describe the behaviour being detected:

```
detection-rules/retail/pos_after_hours_access.kql
detection-rules/retail/gift_card_abuse.kql
```

### Required file header comment

Every rule must open with a comment block containing:

```kql
// Rule:      <Human-readable rule name>
// Severity:  Critical | High | Medium | Low
// Tactic:    <MITRE ATT&CK tactic>
// Technique: <MITRE technique ID, e.g. T1078>
// Author:    Tanvir Farhad
// Purpose:   One sentence describing what this rule detects.
```

### Severity guide

| Severity | When to use |
|----------|-------------|
| Critical | Direct impact on customer data, payment systems, or active ransomware |
| High | Lateral movement, credential abuse, or exfiltration indicators |
| Medium | Reconnaissance, policy violations, or suspicious but ambiguous behaviour |
| Low | Informational detections and baseline anomalies |

### Writing the query

- Target the `SecurityEvent`, `SigninLogs`, `AuditLogs`, `CommonSecurityLog`, or custom table relevant to the detection
- Use `let` statements for thresholds to make tuning obvious
- Add an inline comment wherever a value is not self-explanatory
- Keep the final `project` statement to the columns a SOC analyst actually needs
- Test against sample data before submitting

### Sentinel wrapper

If you are also providing the Logic App / analytics-rule wrapper, place the JSON in `sentinel-rules/` using the same base filename.

## Adding a frontend module

The portal supports pluggable modules. Adding a new one involves two steps.

### Step 1 — Create the module file

Create `frontend/src/modules/YourModule.jsx`. Follow the structure of an existing module (e.g. `DetectionRules.jsx`):

- Accept `{ onBack, nav }` as props
- Use `<TopBar moduleName="Your Module" onBack={onBack} nav={nav} currentRoute="yourroute" />`
- Wrap the page in `<div style={{ minHeight:'100vh', background:'var(--bg)', display:'flex', flexDirection:'column' }}>`
- Use `className="rs-page"` and `className="rs-page-inner"` for consistent padding
- Use the CSS utility classes (`rs-stats`, `rs-two-col`, `rs-table-wrap`) for responsive layouts
- Any table must be wrapped in `<div className="rs-table-wrap">` with an inner `minWidth` so it scrolls horizontally on mobile rather than breaking layout

### Step 2 — Add the tile to Portal.jsx

In `frontend/src/pages/Portal.jsx`, add an entry to the `MODULES` array:

```js
{ id:'yourroute', icon:YourIcon, name:'Your Module', desc:'One sentence description.', color:'#HEX', active:true },
```

Then add the route in `frontend/src/App.jsx`:

```jsx
case 'yourroute': return <YourModule nav={nav} onBack={() => nav('portal')} />;
```

Import the component at the top of `App.jsx`.

## Running the test suite

The test suite covers KQL detection rules using pytest.

### Prerequisites

```bash
pip install pytest pyyaml
```

### Run all tests

```bash
cd detection-rules
pytest tests/ -v
```

Tests validate rule syntax, required header fields, MITRE mapping presence, and severity values. All 221 tests must pass before a PR touching detection rules will be reviewed.

### Frontend

There is no automated frontend test suite at present. Test changes manually at 375px, 768px, 1024px, and 1440px viewport widths before submitting a UI pull request.

## Code style

### KQL

- 2-space indentation
- One blank line between logical sections
- `let` bindings at the top, query body below
- Column names in the `project` statement on separate lines when there are more than four

### JavaScript / JSX

- No build tooling beyond what is already in `package.json`
- Inline styles using the CSS custom properties defined in `global.css` — do not hardcode colour hex values that exist as variables
- No external UI libraries beyond Recharts and Lucide React
- No TypeScript (the project is plain JSX)
- No comments unless the reason for a decision is non-obvious

### General

- No trailing whitespace
- Unix line endings
- Files end with a newline

## PR review process

1. Open the PR targeting `dev` with a clear title and description.
2. Include a **How to test** section in the PR body — enumerate steps a reviewer can follow to verify the change works.
3. All PRs are reviewed by the maintainer ([@TFT444](https://github.com/TFT444)) before merge.
4. Address review comments with new commits — do not force-push a branch that is under review.
5. Squash or fixup commits at the maintainer's discretion before merge.
6. PRs will not be merged until the KQL test suite passes (if detection rules are touched).

## Reporting security vulnerabilities

**Do not open a public GitHub issue for security vulnerabilities.**

Report security issues privately by emailing **tanvirfarhad007@gmail.com** with the subject line `[RetailShield] Security Vulnerability`. Include:

- A description of the vulnerability
- Steps to reproduce
- Potential impact
- Any suggested fix

You will receive a response within 72 hours. Please allow reasonable time for a fix to be developed before any public disclosure.
